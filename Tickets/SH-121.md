📄[SH-121] Especificación: BFF WebSocket a gRPC Stream Pipe (Passthrough)
1. Visión General (Abstract)
El Aegis Neural Kernel (ANK) ahora expone el SirenService, un stream bidireccional gRPC para procesamiento de audio. Sin embargo, los navegadores web no pueden hablar gRPC nativo bidireccional de manera eficiente sin proxies pesados (gRPC-Web no soporta client-streaming de forma nativa en todos los entornos).
La solución arquitectónica es usar nuestro BFF (FastAPI) como un Tubo Pasarela (Dumb Pipe). El frontend enviará chunks de audio binario vía WebSocket, y el BFF los retransmitirá al vuelo hacia el Kernel vía gRPC, devolviendo los eventos del Kernel al frontend en formato JSON.
Objetivo del Ticket: Implementar el endpoint asíncrono en FastAPI y la conexión gRPC en el cliente Python, garantizando cero decodificación de audio en la capa intermedia y propagación natural del backpressure.
2. Diseño de Concurrencia (El Patrón Pasarela)
En FastAPI, un WebSocket no puede leer y escribir simultáneamente en un solo bucle while True sin bloquear uno de los sentidos. Necesitamos un diseño de Multi-Tasking Asíncrono.
Paso 1: Validación Citadel. Al abrir el WebSocket /ws/siren/{tenant_id}?session_key=..., el BFF valida los credenciales.
Paso 2: Generador asíncrono (WS -> gRPC). Se crea una función generadora que hace await websocket.receive_bytes(), empaqueta el binario en un objeto siren_pb2.AudioChunk y hace yield. Este generador se pasa al stub gRPC de Tonic.
Paso 3: Consumidor asíncrono (gRPC -> WS). Mientras el gRPC está enviando datos, el BFF itera asíncronamente sobre las respuestas (async for event in response_stream:) y envía websocket.send_json().
3. Control de Fallos y Backpressure (SRE Focus)
Zero-Processing: Python tiene estrictamente prohibido usar librerías como wave, pyaudio o manipular el payload binario. Los bytes entran y salen intactos. Esto mantiene la latencia en el rango de los <5ms.
Caída del Cliente: Si el usuario recarga la página, el WebSocket lanza WebSocketDisconnect. El BFF debe cancelar inmediatamente la llamada gRPC subyacente (call.cancel()) para liberar el worker en Rust.
Caída del Kernel: Si Rust rechaza la conexión por OOM (RESOURCE_EXHAUSTED), el BFF captura la excepción grpc.RpcError y envía un JSON al frontend: {"error": "Kernel Overloaded"} antes de cerrar el socket limpiamente (Código 1008 o 1011).
4. Criterios de Aceptación (Definition of Done - DoD)
Para considerar este ticket completado, el programador debe:
Actualizar el Protobuf en Shell: Copiar la última versión de siren.proto y compilar los archivos _pb2.py y _pb2_grpc.py en el entorno de Python de la Shell.
Extender AnkClient (bff/ank_client.py):
Implementar el método siren_stream(tenant_id, session_key, chunk_generator).
Este método debe llamar al stub de SirenService y retornar el stream de respuesta.
Implementar Endpoint (bff/main.py):
Crear @app.websocket("/ws/siren/{tenant_id}").
Implementar la lógica de concurrencia para mantener el stream de lectura (WS) y escritura (WS) vivos simultáneamente sin deadlocks.
Resiliencia Comprobada: Bloques try/except explícitos para WebSocketDisconnect y grpc.RpcError. Cero uso de código bloqueante (síncrono) en el endpoint.
