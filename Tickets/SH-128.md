### [SH-128] Especificación: Browser-Side VAD & Push-To-Talk Fallback

#### 1. Visión General (Abstract)
El modo "Microkernel" de Aegis OS carece de VAD nativo en Rust (por la ausencia de C++). El Kernel ahora espera recibir un chunk de audio especial con `format = "VAD_END_SIGNAL"` para saber cuándo procesar la voz acumulada enviándola a la nube. El Frontend debe asumir la responsabilidad de indicar el fin del discurso.

#### 2. Mecanismos de Disparo (VAD vs PTT)
Para dar la mejor experiencia, implementaremos dos modos en el `ChatTerminal.tsx`:
- **Modo PTT (Push-To-Talk):** El usuario mantiene presionado el botón del micrófono para hablar. Al soltarlo, se detiene la grabación y se envía el `VAD_END_SIGNAL`. (Más seguro y con menos falsos positivos).
- **Modo Auto-VAD (Silencio Acústico):** Usando la `Web Audio API` (`AnalyserNode`), el navegador calcula el volumen del micrófono. Si el volumen cae por debajo de un umbral (-45dB) durante más de 1.5 segundos, se envía el `VAD_END_SIGNAL` automáticamente.

#### 3. Payload gRPC (El Trigger)
Cuando la Shell determina que la frase terminó, debe enviar a través del WebSocket (`/ws/siren/`):
```json
{
  "sequence_number": 9999,
  "data": "",
  "format": "VAD_END_SIGNAL",
  "sample_rate": 16000
}
El BFF simplemente reenviará este JSON binario/raw al gRPC, detonando la inferencia en la nube.
4. Criterios de Aceptación (DoD)
Botón de micrófono en ChatTerminal.tsx modificado para soportar onMouseDown / onMouseUp (o Tap-to-Talk).
Lógica de Web Audio API en useAegisStore.ts actualizada para medir el volumen y aplicar un debouncer de silencio (VAD en el browser).
Envío correcto del JSON con VAD_END_SIGNAL al detectar silencio o soltar el botón.
